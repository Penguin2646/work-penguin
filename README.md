# 🐧 work-penguin

> 업무 동료 AI **"펭귄"** — 비개발 직원도 자기 업무를 한 단계씩 업그레이드하는 Claude Code 스킬.
> 자비스처럼 채팅창에 **"펭귄"**이라고 부르면 켜집니다.

러쉬코리아 커머셜 플래닝 팀에서 시작한 사내 업무 업그레이드용 스킬입니다.
**"좋은 도구는 많은데, 내 업무에 뭘 어떻게 써야 할지 모르는"** 동료를 위해 만들었습니다.

---

## 한눈에

- 🎯 **실제 산출물로 끝낸다** — 추상적 조언이 아니라 결과물(리포트·대시보드·문서·스크립트)을 만들어 줍니다.
- 📏 **수준을 진단한다** — 지금 업무가 어느 단계인지 *객관적 축*으로 짚고, 한 칸 위를 처방합니다.
- 🔬 **근거로 검증한다** — "업계 최고"라는 말엔 항상 출처와 신뢰도를 답니다. 단정하지 않습니다.
- 📈 **효과를 측정한다** — 기존 대비 얼마나 나아졌는지 재서 **스코어보드**에 쌓고, 다음에 진짜 효과가 있었는지 추적합니다.
- 🌱 **쓸수록 자란다** — 도구·기록·검증된 지식이 쌓이며 추천이 똑똑해집니다.

## 무엇을 하나요? — 5단계 루프

대화 한 번에 이 루프를 돕니다 (입문자는 **퀵윈 먼저** — 5분 안에 결과물 하나, 진단·측정은 원할 때만):

1. **발굴** — "어떤 게 고민이세요? 요즘 제일 손 많이 가는 업무가 뭐예요?" 로 지금 업무 방식을 인터뷰
2. **진단** — 이 업무가 지금 몇 단계인지(아래 *성숙도 렌즈*), 최상위는 어떤 모습인지 — 자료가 있으면 graphify로 실측까지
3. **처방** — 한 칸 위로 가는 방법 + 쓰면 좋은 (이미 설치된) 스킬 추천·사용법. **단, 과잉은 말립니다**(적정선)
4. **제작** — 말로 끝내지 않고 **실제 산출물**을 만들어 줌 (독립 작업은 여러 에이전트가 병렬로)
5. **측정** — 기존 대비 효과(⏱️시간 / 📊지표 / ✨퀄리티)를 재서 **누적 스코어보드**에 기록 → 다음에 효과를 검증

**고정된 결과물을 찍어내지 않습니다.** 당신의 질문·수준·니즈에 따라 매번 다르게 움직입니다.

## 설계 철학

펭귄을 떠받치는 4가지 원칙입니다 (절대 어기지 않습니다):

1. **점진적 공개** — 내부적으론 성숙도/ROI 구조로 돌지만, 사용자에겐 **쉬운 말**로. "L2→L3", "DiD" 같은 용어로 입문자를 겁주지 않습니다.
2. **적응형 · 무한 확장** — 고정 스크립트가 아닙니다. 수준·니즈에 따라 단계를 건너뛰고·압축·재배치하며, 더 맞는 도구는 **목록 밖에서도 찾아옵니다.**
3. **검증 필수 (전 영역)** — 근거 없는 단정을 하지 않습니다. 진단은 *실측*으로, "업계 최고"는 *체크 가능한 레퍼런스*로.
4. **최소 질문 · 퀵윈 우선** — 한 번에 질문 하나, 알면 건너뛰고, 첫 결과물을 5분 안에. 무거워지면 입문자는 떠납니다.

## 작동 원리 — 핵심 엔진

### ① 성숙도 렌즈 (등급이 아니라 *렌즈*)

업무를 L1(수작업) → L2(문서화) → L3(반자동) → L4(자동·시스템)로 봅니다. **단, 이건 공인 "등급"이 아니라 방향을 보는 "렌즈"**입니다. 단계는 선언이 아니라 **객관적 축**으로 정해집니다:

> 인적 개입 · 재사용성 · 반복 한계비용 · 인사이트 지연 · 오류 통제 · 닫힌 루프

그래서 *"왜 L4야?"* 에 **"이 축들에서 높아서"** 라고 **반박 가능하게** 답합니다. 숫자를 던지지 않고 *"지금 L2~L3 — 템플릿은 쓰는데(재사용 ✓) 데이터는 손으로(인적개입 ✗)"* 처럼 짚습니다.

### ② 적정선 (최대 자동화 ≠ 정답)

목표는 *무조건 L4*가 아니라 **ROI가 가장 큰 칸**입니다. 1년에 한 번 쓰는 보고서에 앱을 만드는 건 **오버엔지니어링** — 펭귄이 *"여기까지가 적정이에요"* 라고 말립니다. 적정선은 **맥락 변수**(빈도·볼륨·재사용·수신자·변동성·오류비용)로 갈립니다. 같은 "보고서"라도 사람마다 종착점이 다릅니다.

### ③ 프론티어 루프 (L4는 천장이 아니다)

L4는 *"현재까지 검증된 최상위"*일 뿐입니다. 이미 최상위 근처거나 *"더 없어?"* 라고 물으면, 펭귄은 그 위(beyond-L4)를 **근거·신뢰도와 함께** 제시합니다. 검증되면 라이브러리에 추가해 **다음엔 baseline**이 됩니다.

### ④ 직무별 검증 라이브러리

전문 직무는 *"업계 최고가 뭐냐"* 를 시장의 **공인 모델**로 근거 댑니다. 11개 직무(세일즈기획·분석·디지털세일즈·마케팅·디자인·회계·인사·영업·CS·물류·재무)의 공인 최상위 모델 + 프론티어를 **3개 이상 독립 소스로 교차검증**해 수록했습니다 — Gartner·BCG·NN/g·Oliver Wight·TDWI·AFP 등. 각 항목엔 **신뢰도 표기**(✅확립 / 🟡신흥 / ⚪미확인). *공통 패턴: 거의 모든 직무의 다음 지평 = Agentic/Autonomous AI.*

### ⑤ 검증 두 루프

- **근거 검증** — 진단은 실측으로, 주장은 레퍼런스로. 공인 모델은 *"유사 사례"로만* 정직하게 인용(권위를 빌리지 않음).
- **효과 검증** — 만든 게 진짜 나았나를 스코어보드 *후속확인*으로 다음 세션에 검증.

### ⑥ 학습 · 복리

검증된 프론티어와 실제 효과가 **파일(렌즈·라이브러리·스코어보드)에 쌓여**, 다음 루프가 더 근거 있게 돕니다. 신경망 재학습이 아니라, *스킬에게 가능한 진짜 학습(복리 축적)* 입니다 — 과장하지 않습니다.

## 무엇이 다른가 (이미 brainstorming·thinking-partner가 있는데)

| | brainstorming | thinking-partner | 🐧 펭귄 |
|---|---|---|---|
| 목적 | 아이디어 → 설계/스펙 | 깊은 사고 촉진 | **실제 업무 업그레이드** |
| 대상 | 만드는 사람 | 탐색하는 사람 | **뭘 할지 모르는 일반 직원** |
| 끝 결과 | 설계 문서 | 인사이트 | **실제 산출물 + 측정된 효과** |
| 수준 진단 | ✗ | ✗ | **성숙도 렌즈 + graphify** |
| 효과 측정 | ✗ | ✗ | **누적 스코어보드 + 후속검증** |

펭귄은 두 도구의 **강점을 흡수·위임**합니다 — SW 설계는 brainstorming으로, 순수 발산은 thinking-partner로 라우팅(깔려 있으면)하거나 직접 수행. **즉 펭귄만 깔아도 품질이 그 아래로 떨어지지 않습니다.**

## 설치

Claude Code가 깔려 있어야 합니다. 워크스페이스의 `.claude/skills/` 안에 이 repo를 넣으면 됩니다.

```bash
# 워크스페이스 루트에서
cd .claude/skills
git clone https://github.com/Penguin2646/work-penguin.git
```

또는 Claude Code 채팅창에 이렇게 부탁해도 됩니다:
> "이 깃허브 스킬 설치해줘: https://github.com/Penguin2646/work-penguin"

## 사용

채팅창에 **펭귄**을 부르세요:

```
펭귄아, 나 매달 보고서 만드는 거 너무 오래 걸려
```

막막하면 이렇게도 됩니다:
```
펭귄아, 나 뭘 개선할 수 있는지 모르겠어   → 직무 프리셋·예시 갤러리로 시작
펭귄은 어떻게 일하는데?                  → 예시 워크플로우를 화면 뷰로 보여줌
내 OO 폴더 한번 봐줘                     → graphify로 현재 상태 진단
```

## 함께 쓰는 도구 — graphify (컴패니언)

펭귄은 진단할 때, 대화가 **실제 업무 폴더**로 향하면 [graphify](https://github.com/safishamsi/graphify)를 꺼내 그 폴더를 지식그래프로 만들어 "지금 상태"를 그림으로 보여줍니다. 미리 안 깔아도 돼요 — **필요해지는 순간 펭귄이 자동 감지해서 설치를 안내**합니다 (PyPI `graphifyy`, 처음 한 번만). graphify가 없어도 펭귄은 대화 인터뷰만으로 진단을 진행합니다(우아한 폴백). _참고: Windows에서 설치 경로가 매우 길면 numpy 의존성이 Long Path 제한에 걸릴 수 있어요 — 짧은 경로에 설치하거나 Long Path를 켜면 됩니다._

## 업데이트 (자가 알림)

펭귄을 부를 때, 새 버전이 올라와 있으면 **"🐧 새 버전이 있어요 — 받을까요?"** 라고 먼저 알려줍니다. *yes* 하면 펭귄이 `git pull`을 대신 해줘요(갱신분은 그 다음 호출부터 반영). 직접 받으려면:

```bash
cd .claude/skills/work-penguin
git pull
```

> 🔒 **안전장치**: 자가 업데이트는 이 폴더가 *자체 `.git`을 가진 독립 clone*이고 origin이 work-penguin일 때만 동작합니다. 워크스페이스에 번들되거나 복사된 경우엔 **조용히 건너뛰어**, 엉뚱한 repo를 pull하는 사고를 원천 차단합니다. (실제 git 서버로 4시나리오 e2e 테스트 통과 · pull만 하고 push는 안 함 → 개인 기록 유출 경로 없음)

## 검증 · 신뢰

펭귄의 신뢰는 검증에서 나옵니다.

- **구성 원리는 입증된 것 위에** — 닫힌 루프(PDCA), 성숙도 사고, AI 생산성 연구(M365 Copilot RCT 등) 위에 설계.
- **주장엔 항상 출처와 신뢰도** — "업계 최고"는 공인 모델로 근거 대되, *"유사 사례로만"* 정직하게. 모르면 모른다고 합니다(환각 금지).
- **효과는 스코어보드가 데이터로** — 검증 장치가 내장된 도구입니다.
- **자가 업데이트도 실측 검증** — 말로 "될 거예요"가 아니라 실제 git 서버로 e2e 테스트 후 배포.
