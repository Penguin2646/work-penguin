# graphify-diagnosis — 실측 진단 (2단계, 제안형)

진단의 **기본은 대화(인터뷰)**다. 하지만 대화가 로컬 자료로 향하거나, 반복·파편화를 정확히 봐야 할 때 → **graphify를 제안**해 사용자의 실제 업무 폴더를 그래프화하고 객관적 근거로 현재 상태를 읽는다. **강제하지 않는다** — 입문자는 대화만으로도 진행된다.

## 언제 graphify를 꺼내나 (트리거)

- 사용자가 "자료가 폴더에 쌓여 있다"는 식으로 말할 때
- 같은 작업을 반복·복붙한다는 신호 → 실제 파편화를 눈으로 확인하면 진단이 정확해짐
- 사용자가 "직접 보여주고 싶다" / "내 자료 좀 봐달라"
- 인터뷰만으론 현재 단계 판단이 애매할 때
→ 이럴 때 한 마디: "그 폴더 한번 열어서 지금 상태를 그림으로 정확히 봐드릴까요? 1~2분이면 돼요."

## 0. graphify는 컴패니언 (자동 감지·설치)

graphify는 별도 스킬/패키지(PyPI `graphifyy`)다. 진단 전 설치 여부를 확인하고, 없으면 안내·설치한다.

- **있으면** 그대로 사용.
- **없으면**: "현재 상태를 정확히 보려면 graphify가 필요해요. 한 번만 깔면 돼요." 하고 한 줄 설치:
  - `uv tool install graphifyy` (uv 있으면) 또는 `pip install graphifyy`
  - 스킬 파일이 없으면 graphify 스킬도 설치 안내(마켓플레이스/GitHub).
- **그래도 불가**: 폴백 — 인터뷰식 진단으로 대체(아래 5). 진단 자체를 건너뛰지 않는다.

## 1. 폴더 받기 (사용자가 동의한 뒤)

graphify를 제안해 동의를 얻으면:
> "그 업무 자료가 **어느 폴더**에 모여 있어요? (예: `10-projects/세일즈리뷰`)"

- 폴더가 흩어져 있으면 대표 폴더 하나부터.
- 민감 자료면 "이건 **로컬에서만** 분석하고 외부로 안 보내요" 안심시킨다.

## 2. graphify 실행

```
/graphify <폴더경로>
```
산출(`graphify-out/`):
- `graph.html` — 인터랙티브 시각화 (브라우저로 열어 보여줌)
- `GRAPH_REPORT.md` — 사람 말로 된 구조 설명
- `graph.json` — 질의용 (`graphify query "..."`로 추가 탐색 가능)

## 3. 그래프에서 "진단 신호" 읽기

graph/report에서 현재 성숙도를 가늠하는 단서:
- **파편화**: 같은 주제 파일이 여러 버전·여러 폴더에 흩어짐 → 구조화 안 됨(낮은 단계)
- **수작업 흔적**: `최종_final_v3.xlsx` 류 중복, 날짜만 다른 반복 산출물 → L1~L2
- **연결 부재**: 입력 데이터와 결과물이 그래프상 끊겨 있음 → 자동화 여지 큼
- **반복 구조**: 매주/매월 같은 모양이 복제됨 → 템플릿·자동화 1순위 후보
- **이미 도구화**: 스크립트·스킬 산출물이 보이면 → 이미 L3 이상

## 4. 사다리에 대입 (진단 = 실측 + 근거)

> "자료를 직접 보니 [그래프에서 본 것]이에요. 그래서 지금은 **[단계]** 같아요. 최상위는 [L4 모습]이고, 바로 위 한 칸은 이렇게 갑니다."

반드시 **그렇게 본 근거**(그래프/리포트에서 관찰한 것)를 함께 말한다. 근거 없는 단정 금지. → `verification.md`

## 5. 폴백 (graphify 불가/폴더 없음)

인터뷰식으로 대체:
- "그럼 말로 같이 그려볼게요." → `flow-guide.md` 2단계 질문으로 Before 스냅샷 작성 → 사다리 대입.
- 폴백이어도 진단 결과엔 "무엇을 근거로 그렇게 봤는지"를 남긴다.

## 6. 시각화 재활용

여기서 만든 `graph.html`은 2단계 시각화로 바로 보여주고, 개선 후엔 다시 graphify를 돌려 **before/after 그래프 비교**도 가능하다.
