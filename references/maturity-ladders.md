# maturity-ladders — 업무 성숙도 렌즈 (seed, 자라는 지식)

> ⚠️ 이건 **등급(grade)이 아니라 방향 렌즈(lens)**다. 공인 자격증이 아니라 *"더 성숙한 방향"*을 보는 실무 도구.
> 누가 "이거 공식 표준이야?"라 물으면 정직하게: **"아니요, 실무 렌즈예요. 단, 단계는 객관적 축으로 정의돼 반박·합의가 가능하고, L4는 천장이 아니라 *현재까지 검증된 최상위*예요."**
> 사용자에게는 "L2/L3"라는 숫자보다 **그 단계의 모습**과 **왜 그렇게 봤는지(축)**를 쉬운 말로.
> 이 파일은 **시드**이자 **자라는 지식**이다 — 검증된 더 높은 방식을 발견하면 여기에 추가한다(아래 프론티어 루프).

## 단계를 가르는 객관적 축 (왜 L4가 L4인가) ★핵심

단계는 *선언*이 아니라 **이 축들에서의 위치**로 정해진다. "왜 L4야?"의 답은 언제나 *"이 축들에서 높아서"*. 그래서 반박·합의가 가능하다.

| 축 | 낮음(L1 쪽) → 높음(L4 쪽) |
|----|---------------------------|
| 인적 개입 (매 사이클 사람 손) | 매번 → 거의 0 |
| 재사용성 | 일회성 → 템플릿 → 시스템·컴포넌트 |
| 반복 한계비용 | 매번 풀비용 → 0에 수렴 |
| 인사이트 지연 | 사후에야 앎 → 실시간 |
| 오류 통제 | 수기 오류 노출 → 자동 검증 |
| 루프 | 스냅샷(끊김) → 환류(닫힌 루프) |

→ 진단은 이렇게: *"지금 L2~L3 사이예요 — 템플릿은 쓰는데(재사용 ✓) 데이터는 손으로 넣고(인적개입 ✗) 사후에야 보거든요(지연 ✗)."* **숫자를 던지지 말고 축으로 위치를 짚어** 반박 가능하게.

## 공통 4단계 (축들의 대표 조합)

| Lv | 성격 | 축 프로파일 |
|----|------|------------|
| L1 | 수작업 · 텍스트 | 전 축 낮음 |
| L2 | 문서화 · 정형 | 재사용·오류통제 일부 ↑ |
| L3 | 반자동 · 스킬활용 | 인적개입 ↓ · 한계비용 ↓ |
| L4 | 자동 · 시스템 | 인적개입 ≈ 0 · 실시간 · 닫힌 루프 |

> **L4 = "천장"이 아니라 "현재 알려진 최상위".** 어떤 축이든 더 밀 수 있으면, 또는 새 축(자율 의사결정·자가 최적화 등)이 생기면 그 위가 있다 → **프론티어 루프**.

## 맥락 변수 — 적정 단계는 사람·업무마다 갈린다 ★★ (약점 보완)

> **회사 안에선 큰 범주(보고·데이터·기획…)가 비슷**해서 아래 도메인 표가 좋은 **baseline(기본값)**이 된다. 하지만 같은 "보고서"라도 세부 맥락에 따라 **적정 종착점이 갈린다.** 펭귄은 표를 *법칙*이 아니라 *출발점*으로 쓰고, 아래 변수로 **그 사람에게 맞는 칸**을 도출한다.

**적정선 원칙 (절대 어기지 말 것)** — 목표는 *최대 자동화(L4)*가 아니라 **ROI가 가장 큰 칸**이다:
- 1년에 한 번 쓰는 보고서 → **L2(PDF)가 적정.** 앱/웹은 **오버엔지니어링.**
- 매일 보는 실적 → 그제서야 **L4(대시보드)**가 산다.
- *"더 위로 갈 수 있다"* ≠ *"더 위로 가야 한다"*. 과잉이면 펭귄이 **"여기까지가 적정이에요"**라고 말린다. (이게 신뢰의 핵심)

**적정선을 가르는 변수** (높을수록 위 칸이 ROI가 산다):

| 변수 | 낮은 목표 쪽 ← → 높은 목표 쪽 |
|------|------------------------------|
| 빈도 | 일회성·연간 → 주간·일간·실시간 |
| 볼륨 | 소량 → 대량 |
| 재사용 지평 | 버리는 것 → 반복되는 것 |
| 수신자 | 본인 → 팀 → 임원·외부 |
| 변동성 | 양식 고정 → 자주 바뀜 |
| 오류 비용 | 낮음 → 높음(돈·신뢰 직결) |

→ 적용: 막연히 L4를 밀지 말고 **핵심 변수 한두 개만 묻는다** — *"이거 매일 하세요?(빈도) 받는 분이 누구예요?(수신자)"* → 그걸로 **적정 칸**을 잡고, *왜 그 칸인지*를 변수로 설명한다.

**세분화 & 학습 (변수가 갈래를 만든다)** — 1+1=2처럼 다 박을 순 없으니 *변수로 갈라지게* 둔다:
- 같은 카테고리도 세부가 갈리면(예: "보고" 안에 *연간 기획보고* vs *일일 실적보고*) **서로 다른 적정선**을 가진다 → 펭귄이 변수로 구분.
- 새 갈래 + 그 **검증된 적정선**이 나오면 → **이 파일에 하위 갈래로 추가**(변수 프로파일 + 적정 목표 + 근거). 다음에 같은 갈래가 오면 baseline이 된다.
- 효과는 스코어보드로 환류 → 그 갈래 신뢰도 조정. → **표가 "납작한 4단계"에서 "변수로 갈라지는 결정 구조"로 자란다.**

## 공인 모델 (뿌리 아님 — "유사 사례"로만 인용) ★정직

이 렌즈를 *발명*한 건 아니다. "수작업→시스템화" 패턴은 여러 모델에 걸쳐 관찰된다. **단, 아래는 "출처/근거"가 아니라 "유사 모델"로만 댄다** — 각 모델은 원래 *다른 걸* 재므로 겹치는 곳에서만:

| 우리 렌즈 | 유사 모델 | 원래 재는 것 (겹침 주의) |
|-----------|-----------|--------------------------|
| 데이터 분석 (보고→진단→예측→처방) | **Gartner Analytic Ascendancy Model** | 분석 유형 — *진짜로 잘 맞음* |
| 자동화·프로세스 성숙 | **CMMI** (Carnegie Mellon SEI) | 조직 *프로세스 규율* — 우리 "업무 자동화"와 결이 달라 **느슨한 유비만** |
| 데이터 조직 성숙 | **TDWI Analytics Maturity Model** | 조직 *도입 수준* |

→ 회의적 질문(*"CMMI Level 4는 그 뜻 아닌데?"*)에 정직하게: **"맞아요, CMMI는 조직 프로세스 모델이라 우리 건 느슨한 유비예요. 우리 단계의 진짜 근거는 위의 *객관적 축*이에요."** 모델명으로 권위를 빌리지 말고, 축으로 방어한다.

## 🔁 프론티어 루프 (이 스킬의 핵심 — beyond L4 + 학습) ★★

L4에 도달했거나 사용자가 *"더 없어?"*라 물으면 **"끝"이라 하지 않는다.** 검증된 더 높은 프론티어를 찾아 제시하고, 결과를 다시 학습한다.

1. **위치 진단** — 축으로 현재 위치 (반박 가능하게).
2. **현재 최상위 제시** — 이 파일에 기록된 *검증된* L4.
3. **프론티어 스캔** — 그 위를 능동 탐색: *어떤 축을 더 밀 수 있나? 새 축이 생기나?* 근거는 **WebSearch / research-worker로 실제 확인.** 제시는 **3종 세트 + 신뢰도**:
   - **무엇이** 더 고도화된 모습인지 (구체)
   - **왜** 더 성숙인지 = **어떤 축을 미는지**
   - **체크 가능한 출처** (사례·회사·프레임워크)
   - **신뢰도 표시**: ✅ 확립된 실무 / 🟡 신흥 / ⚪ 실험적·미확인
   - 근거를 못 찾으면 정직하게: *"여기까진 확인됐고, 그 위는 아직 제가 확인 못 했어요."* — **환각 금지.**
4. **방향 유도** — 점프 강요 말고 **다음 한 칸**을 축 기준으로 구체적으로.
5. **환류·학습 (루프를 닫는다)** ★:
   - 검증된 새 프론티어 → **이 파일에 새 단계(L5…)나 새 축으로 추가 제안** (출처와 함께). 다음엔 그게 "현재 최상위". → **사다리가 자란다.**
   - 추천 방향의 **실제 효과** → 스코어보드 후속확인으로 검증. 효과 있으면 그 방향 신뢰도 ↑, 없으면 원인 보고 축 재조정. → **추천이 똑똑해진다.**
   - 도메인 인사이트는 wiki synthesis로 축적(복리). → `verification.md` · `measurement.md`

> **정직한 "학습"의 정의**: 이건 모델 재학습이 아니다. **검증된 프론티어와 실제 효과가 파일(사다리·스코어보드·wiki)에 쌓여, 다음 루프가 더 근거 있게 도는 것** — 스킬에게 가능한 진짜 학습(복리 축적)이다. **과장하지 않는다.**

---

> 아래 도메인별 표는 **baseline(회사 공통 기본값)**이다. *법칙이 아니다.* 실제 적정 칸은 위 **맥락 변수**로 조정하고(과잉 금지·ROI 피크), 표에 없는 업무는 *축*으로 즉석 생성한다. 가치 있는 새 갈래는 추가(학습). 각 L4는 *현재 최상위*이며 프론티어 루프로 갱신될 수 있다.
> 📚 **전문 직무(마케팅·디자인·회계·인사·영업·CS·물류·재무·세일즈기획·분석·디지털세일즈)의 검증된 공인 최상위 모델·출처·beyond-L4 프론티어 → `domain-frontiers.md`** (11개 직무, 3소스 교차검증, 복리로 자람).

## 📈 세일즈 기획 (Sales Planning)

> 핵심 통찰: 대부분 "얼마 팔겠다 / 뭘 만들겠다"는 **1~2차원 목표**에서 멈추고, *잘 팔리면 좋고 아니면 말고* 식으로 **스냅샷처럼 끊긴다.** 최상위는 결과의 강점·약점·리스크를 진단해 **다음 계획으로 환류하는 닫힌 루프**다 (= 위 "루프" 축이 최대).

| Lv | 모습 | 예시 도구 |
|----|------|-----------|
| L1 | 1~2차원 목표만 ("X개 팔자") · 사후 회고 없음 | — |
| L2 | 목표 + 사후 실적 비교 (팔렸나/아니냐 기록) · 여전히 스냅샷 | product-forecast, excel-to-csv |
| L3 | 결과의 **강점·약점 분석 + 리스크 진단** 추가 (왜 됐/안 됐나) | sales-analysis, pm-skills(SWOT·Five Forces·리스크) |
| L4 (현재 최상위) | **닫힌 루프**: 계획→실행→진단(강약점·리스크)→환류→재계획이 끊김 없이 반복 · 예측·실시간 모니터링 | product-forecast + sales-analysis + graphify + 대시보드 |

**세일즈 기획에서 펭귄이 하는 일** = 단발 산출물이 아니라 **업무를 루프로 재설계**:
1. 목표를 다차원으로 (무엇을·얼마·왜·리스크는)
2. 실행 후 강점·약점·리스크를 구조적으로 진단 (축·레퍼런스로 검증)
3. 그 진단을 다음 계획에 환류 → 스냅샷을 루프로 연결

> **근거·프론티어** → `domain-frontiers.md` (Gartner 5단계 S&OP·Oliver Wight IBP ✅, beyond-L4 = Autonomous Planning·Agentic 🟡)

---

**진단 원칙**: 현재 위치를 **축으로** 짚고 → *현재 최상위(L4)*를 보여주고 → **바로 위 한 칸**만 구체적으로 제시한다. 입문자에게 L1→L4 점프를 강요하지 않는다.

**프론티어 원칙**: L4는 천장이 아니다. 도달했거나 사용자가 더 원하면 **검증된 프론티어(beyond L4)를 근거와 신뢰도와 함께** 제시하고, 검증되면 이 파일에 더한다.

**루프 원칙**: 많은 업무의 최상위는 단발 산출물이 아니라 **닫힌 루프**다. "스냅샷처럼 끊긴 업무를 루프로 잇는 것"이 디벨롭의 본질 — 펭귄은 산출물만이 아니라 *업무의 구조 자체*를 루프로 재설계하고, 그 결과를 학습해 사다리를 키운다.
